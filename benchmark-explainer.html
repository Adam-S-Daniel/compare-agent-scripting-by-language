How this benchmark works

A 5-minute orientation before you read any blog post about the numbers.

The one-sentence version We give a Claude Code agent a small DevOps scripting task, force it to write the solution in a chosen language (or let it pick), watch it build and test the thing inside a real GitHub Actions container, and grade the result on speed, cost, and quality.

What gets measured

Every (language × model+effort) cell of the report has four headline columns:

Duration

Wall-clock minutes per run. Mean across the 7 tasks.

Cost

USD per run from the CLI's own token accounting. Mean across the 7 tasks.

Tests Quality

Panel-LLM score 1–5 of the test code. Coverage / rigor / design / overall.

Workflow Craft

Panel-LLM score 1–5 of the deliverable (YAML + scripts, not tests).

How tiers work

Each headline column is normalised against the best cell observed, then sliced into 13 letter bands. A+ is the best in the run; F is reserved for ratios worse than the worst observed. Bands are auto-calibrated — an A+ in one report is not directly comparable to an A+ in a different report.

How quality is judged

Two LLMs — Haiku 4.5 (via Claude CLI) and Gemini 3.1 Pro (via Gemini CLI) — independently score every run on a 1–5 rubric. The cell shown in the report is the mean. A separate Judge Consistency section flags rows where the two judges disagree by ≥ 4 points and audits their rationales against the actual generated files; if a judge made a false claim it gets dropped from that cell.

So when a blog post says "Workflow Craft B+", that is — literally — the average of two LLM opinions, sanity-checked.

Concepts that show up in posts

Traps

Heuristic detectors for known agent anti-patterns: running act push more than twice, 4+ identical test reruns, fixture rewrites, TypeScript type-error fix cycles, mid-run module restructuring, and so on. Each detection is converted to estimated wasted seconds and dollars. The "Avg Duration Net of Traps" column reads as "how fast it would have been without the self-inflicted detours".

Hooks

PostToolUse hooks run a syntax / type / lint check after every Write or Edit. If the check fails, the diagnostic is fed back to the agent before its next turn. The Hook Savings table is a counterfactual: if the hook caught an error, that's one test cycle the agent didn't have to spend. tsc --noEmit is the most active hook by far.

Default = Python

In the unconstrained default mode the agent picks its own language. In the latest run it picks Python ~98% of the time, so any "default vs forced" comparison is essentially "Python vs the forced language."

Where the raw numbers live

Per-run artefacts in results/<date>/tasks/<task>/<variant>/: full CLI JSON stream, console transcript, metrics.json, before/after workspace listings, and a copy of every file the agent generated. Aggregated tables: results/<date>/results.md.

Caveats worth keeping in mind

  • n is small. 7 tasks per cell. A single bad run moves the mean visibly.
  • Cost is what the CLI reported, not a third-party meter. Cache reads count cheap.
  • Tiers are relative to this run. Re-calibrated each report; do not compare A+ across runs.
  • The agent's own tests can lie. "tests pass" means the agent's tests passed — not that the script is correct. Tests Quality is the partial answer to that.
  • Judges have biases. They tend to dock structural counters they can't see (low coverage scores despite many tests). The Judge Consistency section is the place to verify before quoting a number.
