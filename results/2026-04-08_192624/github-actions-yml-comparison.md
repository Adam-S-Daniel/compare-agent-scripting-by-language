**Run:** `2026-04-08_192624` (v3 benchmark, 64 runs: 8 tasks x 4 modes x 2 models)

## 2. Language-Specific Patterns

### Default Mode (agent chose Python in all 16 runs)

- **Script language:** Python (external `.py` scripts), always via `python3 script.py`
- **Runtime setup:** Opus uses `actions/setup-python@v5` in 6/8 tasks; Sonnet only 3/8 (relies on system Python)
- **Testing in workflow:** Sonnet runs `pytest` in all 8 workflows; Opus only 1/8
- **Size:** Smallest workflows overall (Opus avg 82 lines = most minimal of all)
- **Job structure:** Opus favors single-job workflows (avg 1.8 jobs); Sonnet uses multi-job with `needs:` chains (avg 3.0 jobs including test fixture jobs)
- **Trigger style:** Sonnet tends to use `"**"` (all branches) + schedule; Opus targets `main`/`master` only

**Opus example (task 17, 35 lines):** checkout -> setup-python -> validate input exists -> run script. No tests, no schedule, no frills.

**Sonnet example (task 12, 220 lines):** run-tests (pytest) -> assign-labels -> 4 parallel fixture test jobs (docs-only, API, mixed, no-match), each with grep assertions.

### Bash Mode

- **Script language:** External `.sh` scripts (e.g., `./script.sh`)
- **Validation:** Sonnet includes `shellcheck` in 6/8 workflows + `bash -n` syntax checks; Opus only 2/8
- **Testing in workflow:** Sonnet uses `bats-core` in 5/8; Opus only 1/8
- **Size:** Largest workflows of any mode (avg 155-156 lines for both models)
- **Job structure:** Both use fan-out patterns. Opus tends toward matrix strategies or separate dedicated jobs per test scenario (e.g., task 18: validate -> 6 parallel test jobs -> summary)
- **Unique traits:** Only mode where both models produce comparably large workflows. Bash inherently requires more inline code for assertions, making workflows verbose.

### PowerShell Mode

- **Script language:** External `.ps1` scripts
- **PowerShell installation:** Opus installs pwsh manually in 6/8 workflows (checking `command -v pwsh` first); Sonnet installs via `apt-get` in all 8/8
- **Shell directive:** Opus uses `shell: pwsh` in 3/8 (mixes bash for setup + pwsh for logic); Sonnet uses it in 4/8
- **PowerShell escaping problem:** When using `run:` blocks in bash shell to invoke `pwsh -Command`, Sonnet sometimes nests PowerShell code inside bash strings, requiring `\$` escaping -- fragile and hard to read
- **Testing:** Both use Pester (Opus 4/8, Sonnet 5/8)
- **Size:** Opus avg 86 lines (compact single-job workflows), Sonnet avg 120 lines
- **Unique traits:** Only mode requiring runtime installation. This overhead is reflected in the benchmark data -- powershell/sonnet was the slowest mode overall (21.0min avg) and triggered `docker-pwsh-install` traps. Sonnet's powershell-sonnet is the only mode-model to use `actions/upload-artifact@v4` (for Pester NUnit XML results)

### TypeScript-Bun Mode

- **Script language:** External `.ts` scripts run via `bun run script.ts`
- **Runtime setup:** Opus uses `oven-sh/setup-bun@v2` in 6/8; Sonnet in 7/8 (highest adoption of official action)
- **Testing in workflow:** Sonnet runs `bun test` in 7/8 workflows; Opus only 2/8
- **Size:** Opus avg 86 lines (tied for smallest with default-opus and powershell-opus); Sonnet avg 109 lines
- **Unique traits:** Some Sonnet workflows install `actionlint` from GitHub releases and lint the workflow itself. TypeScript mode also had the highest hook catch rate (53% for Opus, 47% for Sonnet) due to `tsc --noEmit` catching type errors early.

## 3. Model Comparison: Opus vs Sonnet

### Opus Tendencies
- **Minimalist:** Smaller files, fewer jobs, fewer steps
- **Single-job preference:** avg 1.9 jobs/file across all modes
- **Fewer in-workflow tests:** Relies on external test runs rather than embedding test assertions in workflow steps
- **Targeted triggers:** Prefers `main`/`master` branch filters
- **Top-level env blocks:** Uses top-level `env:` more often (14/32 vs 3/32 for Sonnet)
- **dispatch inputs:** Defines `workflow_dispatch` inputs in 18/32 workflows (56%)
- **Less defensive:** Rarely installs shellcheck, rarely runs linters in workflow

### Sonnet Tendencies
- **Elaborate:** Larger files, multi-job DAGs with `needs:` chains
- **In-workflow testing:** Embeds test frameworks (pytest 8/8, bats 5/8, bun test 7/8, Pester 5/8) directly in workflow steps
- **Broad triggers:** Uses `"**"` branch patterns + `schedule:` cron in 19/32 workflows (59%)
- **Fixture test jobs:** Often creates separate parallel jobs for different test fixtures (e.g., "Test: docs-only PR", "Test: API PR") with inline grep/assertion validation
- **More GHA features:** More likely to use `GITHUB_STEP_SUMMARY`, `upload-artifact`, `PR types: [opened, synchronize, reopened]`
- **Defensive coding:** Runs shellcheck, actionlint, and linters in workflow steps before executing main logic

## 4. Cross-Cutting Observations

### Universal Patterns (all 64 files)
- **`actions/checkout@v4`** -- used in every single workflow, every job
- **`ubuntu-latest`** -- every single job runs on ubuntu-latest (no macOS, no Windows, no self-hosted)
- **`permissions:` block** -- present in 64/64 workflows; `contents: read` is the overwhelmingly common setting
- **External scripts** -- all modes use external script files; no agent kept all logic inline in the YML

### Trigger Patterns
- `push` + `pull_request` + `workflow_dispatch` = baseline for all workflows
- `schedule:` is the optional extra -- Sonnet adds it 59% of the time vs Opus 22%
- Sonnet uses `"**"` (all branches) more freely; Opus restricts to `main`/`master`
- Only `pull_request` with `types: [opened, synchronize, reopened]` appears in some Sonnet powershell/default workflows

### Test Integration in Workflows
This is the strongest differentiator between models:

| Test Framework | In Workflow (Opus) | In Workflow (Sonnet) |
|---------------|-------------------|---------------------|
| pytest | 1/8 | **8/8** |
| bats | 1/8 | **5/8** |
| Pester | 4/8 | **5/8** |
| bun test | 2/8 | **7/8** |

Sonnet treats the workflow as a comprehensive CI pipeline (lint -> test -> run). Opus treats it as a minimal execution wrapper.

### Runtime Installation Overhead
- **PowerShell:** Both models install pwsh via apt-get/wget (6-8/8 workflows). This is the heaviest runtime dependency -- the benchmark showed `docker-pwsh-install` trap consuming 7.5min total.
- **Python:** Already present on `ubuntu-latest`; `setup-python` is optional polish.
- **Bun:** Quick install via `oven-sh/setup-bun@v2` (most used) or `curl -fsSL https://bun.sh/install | bash` (fallback).
- **Bash:** Zero installation needed. This contributes to bash being a fast mode.

### Matrix Strategies (rare)
Only 5/64 workflows use `strategy.matrix`. When used:
- bash-opus (task 11): 5 test cases in a matrix
- default-opus (task 14): single matrix
- ts-bun-opus (tasks 11, 14): 8 fixture test cases in matrix
- ts-bun-sonnet (task 13): 3 fixtures parameterized via matrix with `fail-fast: false`

Sonnet prefers explicit parallel jobs over matrix (more descriptive job names but more YAML), with the lone exception above.

### PowerShell Installation Methods
Opus and Sonnet consistently differ in how they install pwsh:
- **Opus:** Downloads a tar.gz from GitHub releases, extracts, and symlinks to `/usr/bin/pwsh` -- manual but idempotent
- **Sonnet:** Uses the standard Microsoft APT repo (`packages-microsoft-prod.deb` + `apt-get install powershell`) -- conventional but heavier

This split holds across tasks 11-18.

## 6. Summary

**By language mode:**
- **Default (Python):** Most concise workflows. Language doesn't require runtime installation. Agent keeps workflow simple because Python's stdlib handles most logic.
- **Bash:** Largest workflows because assertions and logic must be inline shell. Both models produce similarly verbose files. Highest use of shellcheck and syntax validation.
- **PowerShell:** Installation overhead is the defining challenge. Workflows split between bash (for setup) and pwsh (for logic). Most fragile due to escaping issues.
- **TypeScript-Bun:** Clean workflows with good tooling support (`oven-sh/setup-bun@v2` + built-in `bun test`). Highest hook catch rate. Compact like default but with explicit typing benefits.

**By model:**
- **Opus:** "Get it done" -- minimal workflows, single jobs, targeted triggers. Trusts the external test suite to validate correctness.
- **Sonnet:** "CI best practices" -- multi-job pipelines, in-workflow testing, schedule triggers, defensive linting. Produces workflows that look like real production CI but at 1.4x the size and 1.7x the cost.

The ideal approach likely falls between the two: Opus's efficiency with Sonnet's testing integration.
